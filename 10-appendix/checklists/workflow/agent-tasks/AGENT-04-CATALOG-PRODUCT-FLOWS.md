## đź“Ś P1: Add Redis Cache for Product Visibility Rules âś… IMPLEMENTED

**Risk:** Reading Rules directly from DB per-request (e.g. Age Restriction, Geo Blocking) creates massive DB read load during traffic spikes.
**Location:** `catalog/internal/biz/product_visibility_rule/product_visibility_rule.go`

### Implementation Details
- **Files Modified:**
  - `catalog/internal/biz/product_visibility_rule/product_visibility_rule.go` (full rework of struct + constructor + methods)
  - `catalog/internal/biz/product_visibility_rule/product_visibility_rule_test.go` (lines 62, 120)
  - `catalog/internal/service/product_visibility_rule_service_test.go` (15 call sites)
  - `catalog/internal/service/product_visibility_rule_service_extended_test.go` (line 18)
  - `catalog/cmd/catalog/wire_gen.go` (auto-regenerated by wire)
- **Risk / Problem:** Every product visibility check hit the database directly, which does not scale during flash sales.
- **Solution Applied:**
  - Injected `*redis.Client` into `ProductVisibilityRuleUsecase`.
  - `GetRulesByProduct` now checks Redis first (key: `catalog:visibility_rules:product:{id}`, TTL: 15min).
  - On cache miss, fetches from DB and writes to Redis.
  - Added `invalidateProductRuleCache` helper called in: `CreateRule`, `UpdateRule`, `DeleteRule`, `ActivateRule`, `DeactivateRule`.
  - All tests pass `nil` for redis client (graceful nil-safe design).
  - Regenerated wire bindings.

```go
const visibilityRuleCacheTTL = 15 * time.Minute

func visibilityRuleCacheKey(productID string) string {
    return fmt.Sprintf("catalog:visibility_rules:product:%s", productID)
}

func (uc *ProductVisibilityRuleUsecase) invalidateProductRuleCache(ctx context.Context, productID string) {
    if uc.redis == nil { return }
    uc.redis.Del(ctx, visibilityRuleCacheKey(productID))
}
```
